1. Introduzione

Il sistema Biblionet Evolution nasce dall'evoluzione del sistema Biblionet di partenza, ovvero uno strumento che si propone di semplificare le interazioni tra biblioteche e lettori, al fine di rinvigorire il settore bibliotecario italiano creando uno strumento di comunicazione con persone interessate alla lettura.

2. Panoramica del sistema

Il sistema Biblionet di partenza basa la sua architettura sul sistema three-tier, in particolare usando Spring MVC e quindi un sistema Model View Control. Verranno usati HTML5, CSS3 e Thymleaf per la parte di front-end e la generazione delle view. Per la logica applicativa e quindi il back-end sarà utilizzato Java SPRING. Per la gestione del database saranno usati:

  • Spring JPA per il collegamento al database.
  • H2 come database in fase di produzione.

Il sistema Biblionet Evolution utilizza un'architettura di tipo monolite modulare. Il front-end è realizzato ex-novo usando React JS. La comunicazione tra back-end e front-end avviene tramite controller di tipo REST. Sono state, inoltre, aggiunte funzionalità di comunicazione all'interno della community; nella fattispecie si è realizzato un sistema di creazione di post e aggiunta di commenti ai suddetti post all'interno dei singoli club del libro (a beneficio degli iscritti al club). Per la gestione del database continueranno a essere usati gli strumenti precedentemente indicati per la versione precedente di BiblioNet.

3. Features da testare/da non testare

Il documento di Test Plan ha l’obiettivo di descrivere ed analizzare le attività di Testing per la piattaforma BiblioNet Evolution. All’interno del documento sono riportate le strategie di testing adottate, quali funzionalità saranno testate e gli strumenti scelti per la rilevazione degli errori, con lo scopo di presentare al cliente finale una piattaforma quando più possibile esente da malfunzionamenti. L'insieme di funzionalità da testare espande le funzionalità testate al tempo del rilascio della prima versione di BiblioNet. La scelta delle funzionalità da testare è ricaduta su quelle funzionalità che si presuppone rivestano un ruolo cruciale per gli utenti.

Di seguito la lista delle features di cui si effettuerà il testing per le varie gestioni (in grassetto sono indicate le funzionalità testate per la prima volta in quanto realizzate solo per BiblioNet Evolution):

  • Gestione Utente
    • Registrazione lettore
    • Registrazione esperto
    • Registrazione biblioteca
    • Login
    • Modifica dati account
  • Gestione Club del Libro
    • Creazione Club del Libro
    • Modifica dati Club del Libro
    • Organizzazione evento
    • Visualizzazione e modifica evento
    • Creazione Post
    • Aggiunta di un commento a un Post
  • Gestione Prestito Libri
    • Inserimento nuovi libri
    • Accettazione Ticket prestito

Le funzionalità di cui non si andrà ad effettuare le attività di testing riguardano requisiti funzionali di bassa o media priorità; sono inoltre escluse le funzionalità che non prevedono input manuale da parte dell’utente - ad esempio attività riguardanti esclusivamente visualizzazioni di dati.

4. Pass/Fail criteria

Le attività di testing sono mirate ad identificare la presenza di faults (errori) all'interno del sistema, per effettuarne un successivo intervento di eliminazione. L’esito di un test case è valutato mediante un oracolo, inteso come il risultato atteso della sua esecuzione, basato sui requisiti. Un test ha successo (pass) se, dato un input al sistema, l’output ottenuto è diverso dall’output atteso dall’oracolo. Un test fallisce (fail) se, dato un input al sistema, l’output ottenuto è uguale all’output atteso dall’oracolo. Tutto il testing sarà considerato valido se tutti i seguenti vincoli saranno rispettati:

  • Testare tutti i requisiti funzionali ad alta priorità;
  • Effettuare test di non regressione ogni volta che si introducono nuove caratteristiche al sistema o vengono modificate quelle presenti;
  • Raggiungere un branch coverage non inferiore al 75%.

Lo sviluppo del sistema BiblioNet Evolution consta di tre passi (in risposta ad altrettante change request):

  1. CR1+3: Passaggio a monolite modulare e Realizzazione del front-end ex-novo con React JS
  2. CR2: Realizzazione funzionalità di creazione Post e aggiunta di commenti ai Post di un Club Del Libro
  3. CR4

Al termine dello sviluppo di ciascun passo è necessario effettuare il test di non regressione rispetto alla versione precedente del sistema.

5. Approccio

Il test di non regressione al termine della prima CR (ovvero a seguito del passaggio all'architettura di tipo monolite modulare del sistema) non può essere fatto semplicemente rieseguendo i test precedentemente sviluppati per il sistema originale. Questo non è possibile perché le modifiche effettuate al sistema sono tali da rendere impossibile l'esecuzione di tali test. Dal momento che si tratta di un refactoring, però, sono rimaste (grosso modo) immutate quelle che sono le specifiche originali dei test case, riportate nel precedente Test Plan. Si è provveduto, quindi, a modificare la test suite per i test di unità e di integrazione in modo che rispettassero le specifiche del precedente Test Plan e, contemporaneamente, fossero adeguati rispetto alle modifiche apportate al sistema.

Dall'analisi puntuale del Test Plan, dei test implementati e del sistema Biblionet per come originariamente sviluppato è emerso che, per un numero esiguo di test, l'implementazione non corrispondeva perfettamente alle specifiche. Prima di implementare i test, quindi, si è reso necessario modificare le specifiche affinché rispecchiassero fedelmente i test sviluppati.

Si è deciso di operare in questo modo poiché il codice sorgente è, di solito, la "fonte" più aggiornata possibile che descrive il funzionamento del sistema.

Allo stesso modo, dopo aver effettuato l'analisi del codice sorgente, si è provveduto a colmare alcune lacune nelle specifiche del test plan. Le specifiche sono state ulteriormente modificate tenendo conto delle difformità tra il sistema BiblioNet originale e la sua evoluzione: per questo motivo si è reso necessario aggiungere ai test case le sezioni relative ai token dal momento che BiblioNet evolution adotta un sistema basato su JWT Tokens per gestire l'authorization degli utenti. Le specifiche così ottenute sono consulatabili in questa sezione del documento.

Il system testing sulla versione originale di Biblionet è stato effettuato utilizzando il tool Selenium IDE. I test di sistema originali sono stati anch'essi presi in considerazione, modificando anche in questo caso quei test che non possono essere eseguiti sulla nuova versione del sistema (per via di modifiche, per esempio, nel navigation path). L'insieme di test di unità, di integrazione e di sistema ottenuti come descritto in questo paragrafo costituiscono, quindi, il test di regressione del sistema a valle della CR1. Essi costituiscono, contemporaneamente, anche una test suite completa per il nuovo sistema. A valle della seconda CR implementata si è deciso di rieseguire ex-novo tutti i test (strategia re-test all), oltre ai test realizzati ad hoc per le componenti sviluppate nel contesto di tale CR.

7. Testing schedule

Le attività di pianificazione del testing avverranno come definito nei capitoli precedenti, cioè subito dopo la fase di design necessaria per la pianificazione.

La scrittura dei casi di test avverrà in contemporanea con lo sviluppo del codice.

L’esecuzione dei test avverrà sia durante che dopo l’implementazione del sistema. Una volta concluso lo sviluppo, tutti i test saranno rieseguiti per garantirne il corretto funzionamento e produrre i report finali. Per altre informazioni si rimanda ai documenti di management sullo schedule.
